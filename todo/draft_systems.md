# Draft Systems Page

Reviewed on 2026-03-18.

Status:
- draft only
- intended to replace the current `/core/` page

## Recommended Title

`Systems`

## Recommended Intro

ALIS should not present itself through a thin feature checklist.

It should present itself through interacting systems:

- survival physiology
- inventory and carrying constraints
- world and loading architecture
- consequence and tradeoff loops
- practical knowledge made meaningful through play

This page should make ALIS feel serious and concrete.

## Survival Physiology

Current public docs describe a physiology-based vitals model built around:

- calories
- hydration
- fatigue
- condition
- threshold states

The important public message is not just "there is hunger and thirst."

The stronger message is:

`ALIS is building survival pressure around grounded physiological logic instead of generic survival bars.`

## Inventory And Carrying

Current public docs describe inventory as a real constraint system, not a bottomless bag.

Publicly documented direction already includes:

- server-authoritative inventory
- weight as a meaningful limit
- item size and spatial fit
- equipment-granted storage
- hands-only baseline when unequipped

Recommended copy:

`In ALIS, carrying capacity is shaped by what you wear, what you hold, and what can physically fit - not just by an abstract item count.`

## World And Loading Architecture

ALIS already documents more than gameplay rules. It also documents how the project is built and delivered:

- Build Service
- CDN
- Launcher
- Orchestrator
- runtime loading

This is worth showing because it signals seriousness, scale, and long-term technical intent.

Recommended copy:

`ALIS treats delivery, boot, and runtime loading as part of the product architecture, not as invisible glue.`

## Meaningful Choice And Consequence

This section should be framed as direction, not overclaimed as finished emergent simulation.

Safe wording:

`ALIS is being shaped around choices that create visible tradeoffs and consequences. The goal is not to drown the player in complexity, but to make decisions matter.`

## Practical Knowledge Through Systems

This is one of ALIS's strongest differentiators and should not be hidden behind the old "educational game" label.

Recommended copy:

`ALIS aims to make practical knowledge useful in play. Real-world logic should help the player reason about food, water, fatigue, equipment, and survival decisions.`

This makes the project distinct without turning it into didactic software.

## Why This Matters

The systems page should make one thing clear:

ALIS is not trying to win attention with a generic feature list. It is trying to build a believable survival world where systems reinforce each other.

That is why the page should connect:

- vitals
- inventory
- equipment
- world structure
- trust-oriented technical architecture

## Recommended Closing

ALIS should sound like a project with real internal structure:

- grounded survival rules
- technical depth
- visible constraints
- room for meaningful choice

That tone is more credible than cinematic marketing copy and more distinctive than another generic survival page.
